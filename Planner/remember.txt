  SECTION 5 — DECISIONS & CONVENTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  These are decisions made during design. Do NOT change these without updating
  this file. Breaking these will cause inconsistencies across the codebase.

  NAMING
  ──────
  - Model files   : PascalCase.schema.js or PascalCase.model.js
  - Controller    : camelCase.controller.js
  - Route files   : camelCase.route.js
  - Service files : camelCase.service.js
  - Job files     : camelCase.job.js
  - Util files    : camelCase.util.js
  - Mongoose vars : PascalCase (UserSchema, CustomerSchema, etc.)
  - Collection names: Plural (Users, Customers, Softwares, Subscriptions, etc.)
  - API routes    : /api/v1/[resource] — always plural, always lowercase

  RESPONSE FORMAT
  ───────────────
  Always use response.util.js — never use res.json() directly in controllers.
  Success: { success: true, message, data, pagination? }
  Error:   { success: false, message, errors? }

  AUTH
  ────
  - Access token: Bearer in Authorization header, expires 15min
  - Refresh token: httpOnly cookie named "refreshToken", expires 7d
  - Never store tokens in localStorage on frontend
  - req.user is attached by auth.middleware after JWT verify
  - req.user shape: { _id, name, role, permissions[], department }

  ROLES
  ─────
  - SuperAdmin: all access, permissions[] array is IGNORED (has everything)
  - Admin: most access, controlled by permissions[] array
  - Standard: restricted, controlled by permissions[] array + data scoping
  - Standard users only see customers where customer.serviceUser === req.user._id

  AUDIT LOGGING
  ─────────────
  - audit.middleware hooks into all POST, PUT, PATCH, DELETE automatically
  - It captures before/after by reading the document before and after save
  - AuditLog is append-only — never add update/delete routes for it
  - performedByRole is snapshotted at time of action (role may change later)

  SOFT DELETES
  ────────────
  - Users: use isActive: false (never hard delete a user)
  - Customers: hard delete allowed for SuperAdmin only
  - Softwares: block delete if active subscriptions exist
  - Subscriptions: status = Cancelled (never hard delete if invoices exist)
  - Invoices: never delete — void/refund instead
  - AuditLog: never delete — ever

  PAGINATION
  ──────────
  Default: page=1, limit=20
  Max limit: 100
  Query params: ?page=2&limit=20&sort=createdAt&order=desc
  All list endpoints must use pagination.util.js — never return unlimited arrays

  FILE UPLOADS
  ────────────
  - All uploads go to Cloudinary — no local disk storage
  - Multer uses memory storage (buffer), then streams to Cloudinary
  - Allowed types: images (jpg,png,webp), documents (pdf,doc,docx)
  - Max file size: 5MB per file
  - Max files per upload: 5

  ERROR HANDLING
  ──────────────
  - All controllers wrapped in try/catch
  - Errors thrown to global error handler in server.js
  - Mongoose ValidationError → 400
  - Duplicate key (11000) → 409
  - CastError (bad ObjectId) → 400
  - JWT errors → 401
  - No permission → 403
  - Not found → 404
  - Server errors → 500

  CRON JOBS
  ─────────
  - All jobs registered in Jobs/index.js — imported once in server.js
  - Jobs use Asia/Kolkata timezone (IST)
  - renewal.job: "0 8 * * *"    (8:00 AM daily)
  - infra.job:   "0 9 * * *"    (9:00 AM daily)
  - uptime.job:  "*/15 * * * *" (every 15 minutes)
  - invoice.job: "0 7 * * *"    (7:00 AM daily)

  SMS SERVICE
  ───────────
  - Primary: MSG91 (India-focused, supports WhatsApp + SMS)
  - All SMS calls go through sms.service.js — never call MSG91 directly in controllers
  - After every send: log result in Communications collection
  - Delivery status is checked async (webhook or polling)

  SECTION 9 — KNOWN ISSUES / WATCH OUT FOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  These are gotchas discovered during development. Read before building.

  GOTCHA-001
    Where : Subscription.schema.js
    Issue : The compound index { customer, software } is NOT unique (unique:false).
            A customer CAN have multiple subscriptions to the same software
            (e.g. they cancelled and bought again). Filter by status=Active
            to get the current one.
    Impact: Do not assume one subscription per customer+software pair.

  GOTCHA-002
    Where : user.schema.js permissions[]
    Issue : SuperAdmin role IGNORES the permissions[] array entirely.
            permission.middleware must check role === "SuperAdmin" first
            and skip the permissions check.
    Impact: Never assign permissions to SuperAdmin — they have all access anyway.

  GOTCHA-003
    Where : AuditLog.schema.js
    Issue : The before/after fields are Mixed type (any shape).
            When logging, do NOT log passwords, tokens, or sensitive fields.
            Strip password from before/after before writing AuditLog.
    Impact: Must sanitize user object before passing to auditlog.service.

  GOTCHA-004
    Where : Invoice.schema.js
    Issue : invoiceNumber must be globally unique and sequential.
            Do NOT generate it with Date.now() or random strings.
            Use invoiceNumber.util.js which queries the DB for the last
            invoice number and increments it. Wrap in a mutex if concurrent.
    Impact: Race condition possible under high concurrency — use DB transaction.

  GOTCHA-005
    Where : Softwares.schema.js delete route
    Issue : A software cannot be deleted if ANY subscription references it,
            even Cancelled ones (for billing history integrity).
            Only allow delete if zero subscriptions reference this softwareId.
    Impact: Add pre-delete check in software.controller.js deleteOne handler.

  GOTCHA-006
    Where : uptime.job.js
    Issue : Software is only marked Broken after 2 CONSECUTIVE failed pings,
            not just one. Store lastFailedAt on the Software or check
            uptimeStatus was already Down before creating a new Broken alert.
            This prevents false alarms from temporary network blips.
    Impact: Check current uptimeStatus before creating new alert.

  GOTCHA-007
    Where : Frontend — Standard role
    Issue : Standard users must only see customers where
            customer.serviceUser === req.user._id.
            This filter must be applied in EVERY customer query in the backend
            when req.user.role === "Standard". Never rely on frontend to hide data.
    Impact: Add role-based data scoping in customer.controller.js getAll handler.

  GOTCHA-008
    Where : SMS Service (MSG91)
    Issue : MSG91 requires DLT-registered templates for transactional SMS in India.
            The TEMPLATE_ID in .env must match the registered template exactly.
            Variable parts are passed as placeholders.
    Impact: Test SMS in sandbox mode first. Production requires DLT approval.

  GOTCHA-009 — ✅ RESOLVED (Session 5)
    Where : Backend/Controllers/auth.controller.js — forgotPassword function
    Fix   : forgotPassword now calls sendResetEmail(user.email, rawToken) from email.service.js
            In production: token is sent by email only — not in response body
            In dev (NODE_ENV !== "production"): if SMTP throws, falls back to body token
            so local testing still works without SMTP configured
